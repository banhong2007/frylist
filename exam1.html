Fast fluxing is when a single FQDN is used in conjunction with many different IP addresses that change relatively rapidly.  In order for this to work the DNS TTL (which describes how long the DNS record should be stored locally) must be set to a low number. It is possible for some legitimate actions to generate DNS queries with a relatively low TTL count however it is very uncommon for this to be coupled with a large number of query responses.  Therefore a combination of a low TTL and a high response rate would indicate fast fluxing.  Recommended numbers are a TTL lower than 300 with an answer count greater than 12. The numbers in the graphs are exaggerated for your convenience.

Alert on match This will perform a query and alert for anything matching the query. If Event ID is 123, then alert.
Alert on match excluding known/authorized criteria The reverse logic can be used for allow lists. For example, alert if the Event ID is not 123.
Repetitive matchesThis will trigger an alert when there are a certain number of matches within a specific time, for example, 50 failed logins in 1 minute.
High fluctuationThis monitors the count of logs matching a query within a certain time period. If the number of logs increases rapidly within the time period, it will generate an alert—for example, if a Windows system starts to generate 50% more events than normal.
Low fluctuationThis monitors the count of logs matching a query within a certain time period. If the number of logs decreases or stops within the time period, it generates an alert. An example of this would be generating an alert if a Windows server stopped sending over logs.
New or changed fieldsThis monitors values within certain fields in a query. If they change or have new values never before seen, an alert is generated. This can be used to monitor Windows service events. For example, if a new service is created with the name of asdlfiuew, it would generate an event so long as that service has never previously been seen.
Aggregation thresholdsThis monitors things such as min, max, average, or count returned by a query. If the returned value deviates from a threshold, it generates an alert. An example of this is the previous change monitoring impact graphs.

Syslog messages sent over network store facility and severity in the computed PRI field

  • PRI is 1–5 characters surrounded by <>
  • Example <189>
  • Example: 189/8 = 23.625 so facility = 23 or local7
  • Facility = PRI/8 rounded down to whole number
  • Severity = PRI - (Facility * 8)
  • Example = 189 - (23 * 8) = 5 so severity = 5 or notice

    In the example shown there are 4 different names for a field that contains the hostname "evil_corp". One strategy to correlate this data more easily is to use the Logstash mutate plugin rename all the fields - Host Hostname server_name and server name - to a common field name such as hostname.

    filter { mutate { rename => { "Host" => "hostname" } rename => { "Hostname" => "hostname" } rename => { "server_name" => "hostname" } rename => { "servername" => "hostname" } }}

    With this code all the logs would use the field called hostname simplifying the search to hostname:evil_corp.

    Applying a tag to each data source that contains logs referencing "evil_corp" might tag unwanted data as well since the data source could contain logs that do not reference the server. Directing logs from the evil_corp server to a specific port and dropping logs are not log enrichment techniques.

    Long tail analysis works best with large amounts of data

        •    Intended for datasets that are similar in nature 
        •    This is why LFO works well with related desktops 
        •    Is why LFO fares poorly with servers (varying purpose)

    While long tail analysis can work across the entire enterprise it works better in common datasets. For instance looking for least frequent occurrences on servers is likely to show many one-off pieces of software. This is because servers are commonly used to host a single service and each server may hold a different and unique service.

    On the flip side looking for least frequent occurrences on desktops tends to work well. To manage and maintain desktops standardizations are usually in place. This means that systems are likely to have similar software patch levels and settings applied.
